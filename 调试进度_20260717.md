# 2ASK光通信系统 调试进度 (2026-07-17)

## 系统概述

- **通信方式**: 2ASK光通信 (LED发射 → 光电接收 → ADC采样)
- **TX端**: STM32G474RE NUCLEO, 85MHz (实际SYSCLK=170MHz, APB1 Timer=85MHz)
- **RX端**: STM32G474RE NUCLEO, 150MHz
- **载波频率**: 50kHz (DAC2 + TIM6)
- **波特率**: 10kHz (TIM2中断, 100us/bit)
- **ADC采样率**: 200kHz (TIM3触发), 每bit采样数 = 20
- **DAC**: 12-bit, zero=纯DC偏移(2400), sinx=DC+50kHz正弦

## 已完成的工作

### 1. RX端编译修复 & 优化
- CMSIS头文件包含顺序: .c文件首行 `#include "main.h"`
- Font multiply defined: 分离Font.h(声明)和Font.c(定义)
- LCD字符重叠: 使用Lcd_fill()清除区域后更新
- 采样率改回200kHz, 码率10kHz, 阈值直接使用12-bit ADC值
- 位同步: 边沿检测 + 中相位采样(每bit第10个采样点)
- 帧解析状态机: FRAME_SEARCH→RECV_TYPE→RECV_LEN→RECV_PAYLOAD→RECV_CRC

### 2. TX端UART通信修复
- **关键发现**: USART1使用PC4/PC5, 不连接ST-LINK VCP
- **修复**: 改用LPUART1(PA2/PA3), 连接板载ST-LINK虚拟串口
- 波特率921600, MicroLib printf重定向(fputc)
- 文件修改: usart.h(usart1→lpuart1), usart.c, main.c, stm32g4xx_it.c, gpio.c(PA5 LED)

### 3. TX端帧发送 & 空闲填充
- 空闲模式: 主循环持续向FIFO推入0x55(01010101)
- 发送帧时: FIFO_Clear() → SendFrame() → tx_frame_pending=1(暂停空闲填充)
- 帧数据发完后FIFO接近空时恢复空闲填充

### 4. 协议栈改进: Break检测
- **问题**: 空闲0x55的bit进入帧解析器导致误检测, 帧数据被0x55淹没
- **解决方案**: 添加break(静默间隔)协议层
  - TX发送帧前先发48个0bit(4.8ms无载波)作为gap
  - RX检测连续0采样超过阈值(8bit=160采样)时进入break模式
  - break模式下bit才送入帧解析器, 空闲0x55被忽略
- **三种break退出机制**:
  1. 帧完成(CRC校验后) → s_in_break=0
  2. 超时(2000bit无帧) → s_in_break=0
  3. (已移除) sync reset机制(发现会打断位同步)

### 5. 串口命令系统
- H: 心跳(返回OK-H)
- Z: 推送裸0x55字节
- G: GAP测试pattern(38.4ms OFF + 38.4ms ON + 38.4ms OFF), 示波器验证用
- D: 编码演示(不实际发送)
- R\<data\>: 发送RAW帧 + 反馈编码详情
- T\<string\>: 发送TEXT帧 + 反馈编码详情

### 6. RX端LCD显示
- THR: 阈值, FRM: 帧计数, BRK: break检测计数
- ONES: 1的百分比, LAST: 最近16bit, BITS: 总bit数
- PAT: 0x55/0xAA pattern计数

## 当前未解决问题

### 核心问题: FRM始终为0 (帧解析失败)

**已验证正常的环节**:
- TX端UART反馈: ✓ (LPUART1, 上位机能收到ACK)
- TX端FIFO→DAC→光输出: ✓ (G命令示波器可见OFF/ON/OFF)
- 光链路: ✓ (TX输出点到RX采样点, 示波器有跳变)
- RX端ADC→阈值判决: ✓ (ONES约50%, 0x55正常检测)
- RX端break检测: ✓ (BRK正常自增, 每次gap触发一次)

**问题定位**: break检测成功后, 帧头0xAAE4未被帧解析器找到

**可能原因分析**:

1. **位同步偏移**: break gap结束后, 边沿检测器在gap结束的0→1跳变时重置相位,
   但由于gap期间s_no_edge_count可能超过阈值导致s_sync_ok=0,
   第一个跳变时虽然s_sync_ok=1但相位可能不对。
   - 需要验证: 在gap=48bit=960采样期间, s_no_edge_count是否超过SAMPLES_PER_BIT*50=1000?
   - 960 < 1000, 所以s_sync_ok应该保持1。但G命令gap=384bit=7680采样, 会超过1000!

2. **帧头0xAAE4的移位检测问题**:
   - 4个sync 0xAA + 帧头0xAA = 5个连续0xAA
   - preamble_sr是16-bit移位寄存器
   - 5个0xAA期间preamble_sr一直是0xAAAA, 不会匹配0xAAE4
   - 0xE4推入后才匹配
   - 但如果采样的bit有偏移, 0xE4可能被采样为错误值

3. **CRC校验失败**: 即使帧头被找到, payload可能有误码导致CRC不通过
   - 需要添加调试: 在CRC失败时也计数(frame_crc_err), 确认是帧头找不到还是CRC错误

4. **break期间gap的0bit也进入了帧解析器**:
   - 当前逻辑: s_in_break=1后, 所有bit(包括gap的0bit)都进入FrameParser
   - gap的48个0bit被移入preamble_sr: 0000...000(48个0)
   - 然后sync 0xAA的bit推入
   - 这应该不影响, 因为0xAAE4的匹配是靠0xE4的独特pattern

5. **边沿检测器在gap期间的行为**:
   - gap全是0, 没有跳变, s_no_edge_count持续增加
   - 如果gap > 50bit(1000采样), s_sync_ok被清零
   - TX的gap=48bit=960采样 < 1000, 所以s_sync_ok保持1
   - 但如果实际gap稍长(比如50+bit), s_sync_ok被清零
   - gap结束后第一个跳变恢复s_sync_ok=1, s_phase=0
   - 这个行为应该是正确的

### 建议的调试方向

1. **添加帧解析调试输出**: 在LCD上显示preamble_sr的值, 确认0xAAE4是否被接近
2. **添加CRC错误计数显示**: 确认是帧头找不到还是CRC错误
3. **验证bit采样正确性**: 在break模式下, 将采样的bit流通过LPUART发送出来, 对比TX发送的数据
4. **减小gap长度测试**: 48bit可能太长, 导致s_no_edge_count超限
5. **增大s_no_edge_count阈值**: `SAMPLES_PER_BIT * 50 = 1000` 可能不够, 改为 `* 100 = 2000`
6. **最根本的验证**: 用G命令的ON段(全1)来验证break后bit采样是否正确
   - ON段应该采到连续的1bit
   - 如果采到0bit, 说明位同步有问题

## 协议格式 (当前)

```
TX发送 (以RTest为例):
┌──────────┬──────────┬──────────┬──────────┬──────┬──────┬──────────┬──────┐
│ 48个0bit │ 4×0xAA   │ 0xAAE4   │ TYPE     │ LEN  │ DATA │ CRC16    │ 回到 │
│ (4.8ms   │ (3.2ms   │ (1.6ms   │ (0.8ms   │(0.8ms│(3.2ms│ (1.6ms   │ 空闲 │
│  无载波) │ 1/0交替) │  帧头)   │  0x01)   │ 0x04)│Test) │ CCITT)   │0x55) │
└──────────┴──────────┴──────────┴──────────┴──────┴──────┴──────────┴──────┘

RX检测:
  idle(0x55) → 连续0采样≥160 → break(帧 incoming)
  → 边沿检测锁定0xAA → 搜索0xAAE4帧头 → 接收帧 → CRC校验
  → 帧完成 或 超时(2000bit) → 退出break → idle
```
